## 1. 目的与适用范围

### 谁该读

- 写新原子概念笔记前（B6 P0 拆分 / B7 P1-P2 / 任意新增）
- 重写已有笔记前（信息密度提升、补章节、补 illustration）
- 任何 reviewer 想知道"达标线"在哪

### 什么文章适用

- **HMM 级原子概念文章**（独立 .md 文件）— 必须满足"必须"项
- **综述 / index 页**— 满足"必须"项的子集（可省 worked example、code）
- **个人笔记 / 文本文档**（`docs/文本文档/`）— 不强制（自由格式）

### 适用粒度判定

参考 [全面升级plan.md §6 概念粒度规则](全面升级plan.md#6-概念粒度规则granularity-rule)。简言之：

> 该概念能教 ≥ 1 小时（含数学+代码+例题）+ 在权威教材里占整个 § + 站内被 ≥ 3 篇引用 → 必须独立成文，按本 spec 11 组件写。

## 2. 11 组件清单（Article Skeleton）

每篇 HMM 级文章按以下 11 组件组织。**"必须"项**全部到位才能 commit。

| # | 组件 | 必须? | 检查标准 |
|---|---|---|---|
| 1 | **Header**（标题 / 1-2 句定位 / 难度 / 前置知识） | ✓ | 含 H1 中文标题 + 副标题英文(italic) + 1 句"这篇讲什么"+ 前置 2-3 篇文章链接 |
| 2 | **直觉引入**（类比 / 物理直觉 / 生活场景） | ✓ | 在任何 formal definition 之前 ≥ 1 段；以小白能懂的类比开头 |
| 3 | **形式化定义**（含中英术语标注） | ✓ | 关键术语用 `**Foo Bar (FB)**` 格式首次标注；定义用 KaTeX 数学符号 |
| 4 | **历史 / 动机** | 推荐 | 1-2 段：谁提出 / 为什么 / 解决什么问题。引文需有年份 + 作者 |
| 5 | **数学推导**（KaTeX，步骤完整） | 视主题 | 数学密集主题必须；推导步骤不省，关键 step 单独成块 |
| 6 | **Illustration**（≥ 1 张 Tufte SVG） | ✓ | 必须 ≥ 1 张符合 §3 规范的 SVG；超过 200 字节的概念应有图 |
| 7 | **伪代码 + Python/PyTorch 实现**（含输出） | 推荐 | 算法类必须；代码可运行；附 print 输出注释 |
| 8 | **Worked example**（具体数值走一遍） | 推荐 | 算法 / 公式密集文章必须；用具体数值演示算法每步 |
| 9 | **Common pitfalls / 易错点对比** | 推荐 | 概念易混的主题必须（如 HMM 的 "Forward vs Viterbi"、batch vs layer norm） |
| 10 | **Related concepts**（前置/后续/横向/竞争） | ✓ | ≥ 2 条跨节链接（前置 + 后续 / 横向 / 竞争方法） |
| 11 | **References** | ✓ | ≥ 3 条；权威教材页码 / 论文 / 课程 / 顶级博客；不引 CSDN/知乎 |

## 3. 写作风格规范

### 3.1 中英混排

- **中文为主**（站内主语言）
- 关键术语首次出现：`**前向算法 (Forward Algorithm, FA)**`
- 英文术语单词大写遵循英文习惯，不要全小写
- 在双语 i18n 模式下，`.md` 中文和 `.en.md` 英文是镜像文件

### 3.2 数学公式

- 全用 **KaTeX**（mkdocs.yml 已配置 `pymdownx.arithmatex`）
- 行内：`$f(x) = ax + b$`
- 块级：`$$ E = mc^2 $$`
- 推导步骤不省。多行公式用 `\begin{aligned}` 对齐：

  ```latex
  $$
  \begin{aligned}
  P(S_t \mid O_{1:t}) &= \frac{P(O_t \mid S_t)\, P(S_t \mid O_{1:t-1})}{P(O_t \mid O_{1:t-1})} \\
  &\propto P(O_t \mid S_t)\, P(S_t \mid O_{1:t-1})
  \end{aligned}
  $$
  ```

- 定义 / 引理 / 定理用 admonition：

  ```markdown
  !!! abstract "Bayes 定理"
      $$ P(A \mid B) = \frac{P(B \mid A) P(A)}{P(B)} $$
  ```

### 3.3 代码

- **语言优先级**：Python > PyTorch > 其他
- **可运行**：避免伪代码 + 真代码混杂；注明导入和依赖
- **附输出注释**：

  ```python
  import torch
  x = torch.randn(2, 3)
  print(x.shape)  # → torch.Size([2, 3])
  ```

- 代码块加语言标识：` ```python `（启用语法高亮）

### 3.4 图示

参考下面 §4 图示决策树。**每篇 HMM 级文章 ≥ 1 张 Tufte SVG**，符合 [全面升级plan.md §3 规范](全面升级plan.md#3-tufte-风格-svg-规范canonical-锁定)。

### 3.5 链接

- **跨节链接** 用相对路径 `[X](../OtherSection/X.md)`，不用绝对路径
- **跨语言链接 不需要手工指 .en.md**：mkdocs i18n 插件自动选择对应语言文件
- **外部链接** 加目标说明：`[arXiv:1706.03762 Attention Is All You Need (Vaswani et al., 2017)](https://arxiv.org/abs/1706.03762)`

## 4. 图示决策树

每张图问 4 个问题，按下面流程选模板：

```
1. 图能展示什么？
├── 节点 + 边的关系（有向 / 无向 / 层次）
│   ├── 隐藏 + 观测的图模型 → graphical_model.svg
│   ├── DP 网格（时间步 × 状态空间） → trellis.svg
│   └── 简单分类树 → comparison_panel.svg 上半部分（根 + 三分支）
│
├── 概念对比（多个并列方案）
│   └── comparison_panel.svg（根 + 三分支 + 三色条卡片）
│
├── 模型架构（数据从输入流到输出）
│   └── architecture_block.svg（主路径竖向 + 残差虚线）
│
├── 历史时间轴 / 模型演进
│   └── timeline_horizontal.svg（水平主轴 + 节点上下交错）
│
└── 上述都不是的特殊几何直观（如算法步骤的可视化）
    └── 手工编排 SVG（参考 HMM HTML §06 forward trellis 风格）
```

### 4.1 SVG 嵌入方式

**Inline SVG（推荐，< 80 行）**：

```markdown
<div class="diagram">
<svg viewBox="0 0 720 280" xmlns="http://www.w3.org/2000/svg">
  ...
</svg>
</div>
<p class="figure-caption">Figure 1 — 一句话直觉总结。</p>
```

**外部 SVG 文件（≥ 80 行 或 跨文章复用）**：

1. 创建 `docs/<section>/<article>/diagrams/figure-N-<name>.svg`
2. Markdown 用 snippets 引入：

   ```markdown
   <div class="diagram">
   --8<-- "<section>/<article>/diagrams/figure-N-name.svg"
   </div>
   <p class="figure-caption">Figure N — 一句话直觉总结。</p>
   ```

   （`base_path` 设为 `docs/`，路径相对 `docs/`）

### 4.2 颜色语义约定

| 颜色 | 变量 | 表示什么 |
|---|---|---|
| 主笔画 | `--dia-stroke` | 中性内容 / 默认笔画 |
| 主强调 terracotta | `--dia-accent` | 关键标注 / 类型 1 |
| Forest green | `--dia-green` | 类型 2 / "积极"或"成功"语义 |
| Deep blue | `--dia-blue` | 类型 3 / "信息"或"输入"语义 |
| Gold | `--dia-gold` | 类型 4 / 数学/公式高亮 |
| Soft stroke | `--dia-stroke-soft` | 辅助 / 注解 |

### 4.3 Tufte 风格的几条铁律

- **少即是多**：不要装饰性边框、阴影、3D 效果
- **直接标注**：边标签直接放在边上（覆盖白底），不用 legend 间接标注
- **混排字体**：中文 / 英文 / 数学 / 等宽 各司其职（中:Fraunces serif；英:同 italic；数学:KaTeX；公式 inline:JetBrains Mono）
- **节点大小有意义**：高亮节点用更大尺寸 / 强调色填充
- **暗色模式自动适配**：所有颜色用 `var(--dia-*)`，dark mode 自动响应

## 5. 质量自检清单（Commit 前过）

写完 / 重写后，按以下清单逐项打勾：

### 内容
- [ ] 11 组件中"必须"项全部到位（Header / 直觉 / 定义 / Illustration / Related / References）
- [ ] 信息密度 ≥ 同节相邻文章（不空话凑字数）
- [ ] 数学推导无 "易得 / 显然" 等跳步词；关键 step 单独成块
- [ ] 代码可运行，附输出注释
- [ ] 跨节链接 ≥ 2 条（前置 + 后续）
- [ ] References ≥ 3 条，包含至少 1 条权威教材

### 图示
- [ ] ≥ 1 张 Tufte SVG（符合 §4 规范）
- [ ] 用 CSS 变量 `var(--dia-*)`，不硬编码颜色
- [ ] 中文标签用 Fraunces / Georgia serif 字体
- [ ] 公式标签用 JetBrains Mono 字体
- [ ] Figure caption 一句话总结放在 `<p class="figure-caption">`

### 双语
- [ ] `.en.md` 同步存在
- [ ] 数学 / 代码 / 图片 alt-text / 路径 不翻译
- [ ] 自然学术英语，非逐字翻译
- [ ] 中英两版结构镜像（H1/H2/H3 一一对应）

### 技术
- [ ] `mkdocs build --strict` 通过（无 warning）
- [ ] `git diff --stat` 重写场景：`+` 行远多于 `-` 行
- [ ] 文件计数：`find docs -name '*.md' ! -name '*.en.md' | wc -l` 不减
- [ ] commit message 写清"新增 / 重写 / 仅路径迁移"

### 拆分（若是 P0 原子概念独立化）
- [ ] 原文件位置保留 1-2 段简介 + 链接，未删除上下文
- [ ] 新文件继承原小节全部内容并扩写
- [ ] 拆分前后总行数显著增加

## 7. `.en.md` 翻译规范（速查）

完整规范见 [CLAUDE.md](CLAUDE.md) 的"国际化 (i18n)"段。要点：

- 自然流畅的学术英语，**非逐字翻译**
- 保留所有 Markdown 格式 / LaTeX 数学 / 代码块 / 图片引用 / 链接
- 图片 alt-text 和文件路径中的中文保持不变
- 不翻译代码片段和变量名
- 保留原始结构（标题层级、列表、表格、admonitions 一一对应）
- 中英两版的 SVG 文件**通常共用**（除非节点标签是纯中文，则需生成 `_zh.svg` + `_en.svg` 两个版本）

## 8. 不做的事（明确排除）

- **不删除任何已有内容**（重写也不允许信息密度下降）
- **不引入装饰性图标 / 表情符号**（除非用户明确要求）
- **不引用未审校博客**（CSDN / 知乎个人答案 / 营销号）
- **不使用 mermaid**（站内已废弃，全部 SVG）
- **不内联 > 200 行 SVG 到 .md**（用外部 SVG + snippets 引入）
- **不在 `<img>` 标签里加载 SVG**（`<img>` 不能解析父页 CSS 变量；用 inline `<svg>` 或 snippets）
